# Cachaza workflow and safety model

Cachaza uses a funnel: broad passive collection first, explicit scope decisions second, and bounded direct validation last. Every stage writes normalized `Finding` objects into the same workspace rather than treating intermediate text files as independent sources of truth.

## Workflow

| Step | Activity | Implementation and boundary |
|---:|---|---|
| 1 | Seed scope | Normalize domains, ASNs, CIDRs, organizations, and exclusions supplied by the operator. |
| 2 | Network context | Resolve domains and correlate BGP Toolkit, RIPEstat, ARIN RDAP, and optional ASNmap evidence. Inferred networks remain candidates. |
| 3 | Passive discovery | Collect CT, provider API, tenant, Subfinder, Shodan, cloud-classification, and archived URL evidence. |
| 4 | Scope preservation | Evaluate each domain, URL, IP, service, and prefix against explicit inclusions and exclusions. |
| 5 | Authorized validation | With `-active`, validate names, certificates, ports, and HTTP services under rate and host-count limits. Automatic Direct-origin validation additionally requires `-authorized`. |
| 6 | Endpoint and WAF mapping | In the full profile, combine GAU history, bounded crawling, JavaScript endpoint extraction, and focused WAF identification. |
| 7 | Reporting | Render the single finding stream into HTML, JSON, TXT, PDF, CSV, artifact lists, checkpoints, and a manifest. |

## Profile boundaries

- `passive` performs no direct application or network scan. DNS and public API lookups can still contact third-party infrastructure.
- `safe` requires `-active` and adds bounded DNS, certificate, port, and HTTP validation.
- `full` requires `-active` and adds historical URLs, bounded endpoint crawling, JavaScript endpoint extraction, and focused WAF identification. Explicit `bypass`, `policies`, and `cve` stages remain outside this profile.
- Explicit active stages selected through `-stages` also require `-active`.

## Scope invariants

- A supplied domain authorizes that domain and its non-excluded subdomains, not unrelated domains found in tenant, CNAME, certificate, or search data.
- A supplied CIDR authorizes addresses inside it, minus exclusions.
- A supplied ASN is an input for network intelligence, but host-count and exclusion rules still apply before direct probing.
- Domain-derived ASNs, prefixes, cloud ranges, CDN addresses, Shodan matches, and historical IPs remain candidates. They do not enter the general active pipeline automatically. When `-origin-auto -active -authorized` is explicitly selected, only IPs generated by the Origin candidate discovery pipeline that pass its correlation, public-address, CDN, third-party, score, candidate-count, per-IP, and global-budget gates may receive the dedicated low-impact Direct-origin validation checks.
- Possible bypasses, secrets, sensitive filenames, policies, and CVEs are labeled as candidates or observations requiring manual validation.

## Evidence contract

Each observation contains:

```text
stage, source, kind, value, in_scope, metadata, observed_at
```

The tuple `(source, kind, value)` is normally deduplicated within a workspace. WAF evidence also includes its target origin in the identity so the same vendor detected on two hosts is not collapsed. Source-specific context is preserved in metadata. `rest/findings.jsonl` is canonical; text lists are derived views.

## Subdomain validation and noise control

`dnsenum` and Fierce discoveries are candidates, not proof that a hostname is a distinct asset. When either bundle is requested alongside the normal active pipeline, discovery runs before DNS and HTTP validation. Cachaza invokes dnsx with root-specific wildcard filtering (`-wd ROOT_DOMAIN`) and passes only dnsx-confirmed names to later port and HTTP stages.

Reports classify subdomains as HTTP-responsive, DNS-only, or unverified. Meaningful HTTP evidence includes 2xx/3xx and authentication or authorization responses such as 401/403; a 404 by itself is not promoted. A status code alone is not sufficient to defeat catch-all hosting because arbitrary names can return the same 200 page or redirect. Wildcard-like and brute-force-only candidates remain in `report.json`, CSV, and `rest/findings.jsonl` for provenance, but they are omitted from executive highlights, the relationship graph, and the PDF appendix.

## Endpoint and WAF boundary

Endpoint inventory is built from GAU, Katana, endpoint-only Cariddi, JavaScript references, and passive URL sources. GAU observations retain their historical provenance and are not evidence that an origin is live. Katana and Cariddi remain within authorized FQDN scope; Cariddi uses `-e -plain` and does not enable `-s` secret hunting. JavaScript analysis records related URLs, routes, API endpoints, and Swagger/OpenAPI/GraphQL references without treating secret-like words as findings.

Nuclei is not an endpoint or vulnerability scanner in Cachaza. The general stage, tagged templates, severity selection, automatic scans, workflows, template directories, and URL lists have been removed. The only permitted invocation uses the immutable `http/technologies/waf-detect.yaml` template with `-rl 1 -bulk-size 1 -c 1 -retries 0`. A second validation layer in the network policy rejects any other Nuclei command.

WAF targets come only from confirmed live HTTP URL findings and are normalized to `scheme + hostname + effective port`. Paths, queries, fragments, credentials, and default ports are removed. A non-standard port survives only when httpx or a crawler confirms a live HTTP(S) URL there. DNS names, Naabu/Nmap services, and unverified historical URLs never become WAF targets. When the WAF stage runs without prior live HTTP evidence, it tries only `https://ROOT_DOMAIN`.

The default WAF tools are `wafw00f,nuclei`. Nmap correlation remains available only when explicitly selected. Reports retain vendor, source, confidence, and target origin, and do not interpret empty or negative output as a detection.

## Failure and resume model

Stages are isolated by default. A source or adapter failure is recorded in the manifest and later stages continue; `-strict` stops at the first failure. Successful stages write cache-keyed checkpoints under `rest/stages/`. A compatible resume loads prior findings and reuses checkpoints whose scope and execution options still match.

Provider-level errors that do not abort the API stage remain visible in `rest/api/provider-status.json`, the HTML report, the run-issues metric, and the terminal summary. A completed stage therefore means orchestration completed; it does not imply every configured provider authenticated successfully.

`-fresh` is deliberately constrained: Cachaza verifies `rest/scope.json`, confirms an identical target specification, and only then resets that workspace.

## Automatic Origin discovery methodology

The `origin` stage uses the following ordered escalation model:

1. Capture a `public_baseline` for the root and responding `www` name. The persisted record contains DNS results, CDN/WAF signals, redirect metadata, status, safe headers, cookie names (never values), title, body hashes, SimHash, observed JS/CSS/favicon paths and hashes, and TLS certificate metadata. Bodies are not persisted unless `-origin-save-bodies` is explicit.
2. Collect Origin candidate evidence from current DNS, already normalized CT/subdomain/Censys/Shodan/urlscan findings, configured VirusTotal historical DNS, bounded SPF/MX analysis, and optional AlterX plus PureDNS/DNSx permutations. Provider wrappers are collapsed into source families, so Uncover-via-Shodan does not count independently from Shodan.
3. Reject invalid, private, reserved, multicast, documentation, known CDN/current-edge, public-resolver, excluded-provider/CIDR, and clearly unrelated third-party addresses. Mail-only infrastructure remains visible but is not selected.
4. Apply deterministic, explainable evidence scores. Strong signals include an exact apex A/AAAA correlation, certificate correlation, urlscan main-document IP, or several in-scope web hostnames. Duplicated signals do not receive duplicated points.
5. Select at most `origin_max_auto_candidates`. `should_auto_validate()` requires the minimum score, public non-CDN addressing, `origin_candidate` classification, sufficient independent/strong evidence, no clear third-party classification, and remaining per-IP budget.
6. In `balanced` or `deep`, and only after `-active -authorized`, use TCP connect on configured web ports, TLS with the target hostname as SNI, and HTTP with the same hostname in `Host`. HEAD is followed by a bounded GET when content is needed. Only `/`, explicitly supplied safe paths, an observed favicon, and a few already observed static resources are eligible.
7. Compare status, redirect chain, stable headers, cookie names, normalized HTML, title, favicon/static hashes, and TLS certificate evidence. When JARM is enabled and TLSX is available, deep mode performs one bounded single-host JARM correlation with exact SNI, concurrency 1, and no retries; a matching fingerprint is supporting evidence, never sufficient by itself. Dynamic values such as dates, request IDs, nonces, cache parameters, and cookie values receive no correlation weight.
8. Stop when `origin_stop_score` is reached unless `-origin-continue-after-match` is explicit. No budget grows automatically.

Initial score bands are `0-29 rejected`, `30-49 related/weak`, `50-64 candidate`, `65-79 probable`, and `80-100 high confidence`. Validation results use `high_confidence_origin`, `probable_origin`, `possible_origin`, `related_infrastructure`, `not_matching`, `not_directly_reachable`, `protected_origin`, `inconclusive`, or `cdn_edge`. Cachaza deliberately does not claim `confirmed_origin`.

The central request budget is persisted in `rest/origin/request-budget.json`; unchanged runs reuse the stage checkpoint, fresh successful validation results remain reusable for 24 hours, and only transient/inconclusive failures are retried. A changed configuration invalidates the stage checkpoint and recalculates selection while retaining prior consumed budget in a resumed workspace.

The module never sends `X-Forwarded-Host`, `X-Original-URL`, `X-Rewrite-URL`, forged client-IP headers, POST requests, authentication, or internal-provider headers. It never runs Nuclei/Arjun/fuzzing against a candidate, scans a whole CIDR/ASN, uses SSRF/proxies, defeats mTLS/AOP/ACLs, or exploits an observed weakness.

### Cloudflare interpretation

- Proxied A/AAAA/CNAME records resolve to Cloudflare anycast addresses; DNS-only records resolve to their actual configured address. A DNS-only web hostname sharing an address with a proxied hostname is therefore strong evidence, but still requires correlation and filtering.
- The official IPv4/IPv6 ranges are cached for seven days and used to reject Cloudflare edge addresses. Cachaza does not download them on every run.
- Authenticated Origin Pulls adds TLS client-certificate authentication. A client-certificate alert or direct 403 with strong passive evidence is `protected_origin` / `not_directly_verifiable`; Cachaza does not try to obtain or spoof a certificate.
- An origin ACL may allow only Cloudflare ranges, producing refusal, timeout, or 403. Strong passive evidence is retained as inconclusive instead of being aggressively penalized.
- Cloudflare Tunnel creates outbound-only connections and may have no publicly routable origin. An empty result is reported honestly rather than inventing an IP.
- Workers, Pages, R2, SaaS, and other serverless architectures may likewise have no dedicated public origin address.

Required artifacts are `rest/origin/public-baseline.json`, `all-candidates.jsonl`, `selected-candidates.jsonl`, `rejected-candidates.jsonl`, `validation-results.jsonl`, `network-classification.jsonl`, `evidence.jsonl`, `request-budget.json`, `final-ranking.json`, `final-ranking.csv`, and `raw/`. Optional limited bodies are stored only under `rest/origin/bodies/`.

## Reporting defenses

- HTML table values are escaped.
- Embedded report JSON escapes HTML-significant code points and script separators.
- HTML includes a restrictive Content Security Policy and disallows network connections.
- CSV values beginning with spreadsheet formula prefixes are neutralized before serialization.
- Reports preserve the distinction between authorized and candidate evidence.
